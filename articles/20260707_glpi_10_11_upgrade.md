# はじめに

GLPI 10.x から 11.x へのアップグレードを FreeBSD / jail 環境で実施した際に遭遇した問題点と、その解決方法をまとめます。

GLPI の公式ドキュメントは Linux を前提とした内容が中心であり、FreeBSD + jail 特有の情報はまず見つかりません。
本記事では、実際のアップグレードで遭遇した「ハマりどころ」を中心に紹介します。

対象環境は以下のとおりです。

- FreeBSD 14.4 (jail)
- nginx
- PHP (pkg)
- MySQL 8.0
- GLPI 10.x → 11.0

---

# 背景

これまでの GLPI 10.x の運用では、FreeBSD の pkg/ports を基本としながらも、GLPI 本体や関連コンポーネントについては、本家リリースとの更新タイミングや運用要件を考慮し、一部を手動で更新・管理していました。

そのため、インストールディレクトリには pkg が管理するファイルだけでなく、運用中に追加・更新したファイルや設定が混在している状態でした。

このような環境で単純に `pkg upgrade` を実施すると、運用中の設定や手動で更新した内容の扱いが分かりにくくなる可能性があります。

そこで今回は、既存環境をそのまま退避した上で新しいパッケージを展開し、必要な設定やデータを段階的に移行する方法を採用しました。

この手順は一般的な pkg の更新手順とは異なりますが、既存環境を保全しながら問題発生時に容易に切り戻せることを重視したものです。

---

# 今回採用したアップグレード手順

## 1. 既存環境を退避

まず、GLPI のインストールディレクトリを退避します。

```sh
mv /usr/local/www/glpi /usr/local/www/glpi10
```

これにより、

- config
- files
- plugins
- marketplace

など、運用中のデータを保護できます。

---

## 2. pkg upgrade を実施

そのまま `pkg upgrade` を実施します。
退避しているため、pkg 管理下のファイルを削除できず `unlink` エラーが表示されますが、インストール自体は継続されます。


---

## 3. 不足していた PHP 拡張を追加

私の環境では以下が不足していました。

- php-tokenizer
- php-bcmath

インストーラ実行時に初めて判明したため、追加インストール後にアップグレードを再開しました。

---

## 4. nginx の設定を GLPI 11 用へ変更

GLPI 11 では公開ディレクトリが `public/` になっています。

GLPI 10 では 動作していた設定のままでは正常にアップグレードできなかったため、公式ドキュメントに沿って

- DocumentRoot を `public/`
- PHP は `index.php` のみ FastCGI でハンドルする

という構成へ変更しました。 

legacy-php サイトでは、 .php を一つの location タグにまとめていたりしますが・・・

```
  location ~ \.php$  {

```

を

```
  location ~ ^/index\.php$ {

```

とすることで、 index.php のみがハンドルされ、 内部で呼ばれる .phpも隠蔽されます。 

---

## 5. Upgrade を実施

ここで Web Installer によるアップグレードを実行しました。

![](/images/2026070701/glpi_setup.png)

データベース更新自体は正常に進みましたが、途中で Permission Error が発生しました。

原因は、以前のアップグレード失敗時に生成された

```
files/_cache
```

配下のファイルが

```
root:wheel
```

所有になっていたためでした。

以下で修正しました。

```sh
chown -R www:www files
```

---

## 6. データベース Migration

アップグレード完了後に

- utf8mb3
- signed integer key

に関する Warning が表示されました。

![](/images/2026070701/glpi_entry.png)

そのため、

```sh
php bin/console migration:utf8mb4
```

```sh
php bin/console migration:unsigned_keys
```

を実行しました。

どちらも完了まで数分程度でした。

---

## 7. 最終確認

最後に

```sh
php bin/console glpi:system:check_requirements
```

を実行し、すべて **OK** となることを確認しました。

さらに

```sh
find files config marketplace plugins \
    -not -user www -o -not -group www
```

を実行し、権限の不整合が残っていないことも確認しました。

---

# ハマりどころ

今回遭遇した問題をまとめると次のようになります。

- jail だったので root で作業してしまう(owner=wwwであるべきものが作られない)
- PHP 拡張が不足していた
- GLPI 11 の nginx 設定へ変更する必要があった
- `files/_cache` の所有者が root のまま(jail+rootで作業した結果)
- `bin/console` が正常に動作せず、pkg の再インストールで復旧した。原因は不明だが owner がrootだったのが関係？
　筆者は vendor/ を管理するのが composer だと思いこんでいましたが、bin/console は独自のshellscriptでした。
- アップグレード後に utf8mb4 / unsigned key Migration が必要だった

---

# おわりに

GLPI 11 へのアップグレードそのものは難しいものではありませんでしたが・・・

一方で、FreeBSD+jail 環境では

- jail で wwwユーザ前提のアプリを扱うこと
- pkg による管理
- 手動で更新したコンポーネント
- Web サーバ設定
- PHP モジュール
- ディレクトリ権限

などが複合的に影響するため、Linux の情報だけでは解決しにくい場面がいくつかありました。

本記事が、同様の環境でアップグレードを行う方の参考になれば幸いです。
