# GitHub Copilot で BDD 事始め

## 1. GitHub を本格的に使い始めた 1 年間の振り返り

### 2025年春まで — まずは慣れる

- 既存のプライベートリポジトリをとりあえず GitHub に集約。
- Copilot に「あれ作って」と頼んでいたが、プロンプトが未熟で出力の質も低く、手直しに疲弊。

### 2025年夏〜秋 — Dependabot との格闘と本格準備

- Dependabot からのセキュリティ通知を受け、PR の扱いを学ぶ。
- Python・PHP プロジェクトの準備を開始。VSCode と統合し、macOS / Linux / FreeBSD / Windows をまたいで作業。

### 2025年10〜12月 — プロジェクト実戦投入

- 業務で 2 プロジェクトを Copilot の助けを借りて稼働させる。
- Python の小規模プロジェクトは準備半年で完封。
- PHP のレガシープロジェクトも難所を越え、ほぼ勝利。

---

## 2. 「TDD なくして Copilot 開発なし」と悟る

### Copilot 初期のつまずき

「あれ作って」式の指示では、次のような問題が繰り返されていた。

- 生成されたコードが「どうも違う」ため、何度も手直しが発生する。
- エージェントは「エラーが出ない」「コンパイルが通る」という最低限の評価しか得られない状態で動くため、**適切な評価基準（= テスト）がなければ質を担保できない**。

### 解決策：「失敗ありき」でエージェントに働かせる

- コードを生成させる前に「これはエラーになる」を前提とした設計をしておく。
- バグ・タイポを早期に検出させ、素早く修正させるサイクルを回す。
- ネットワーク遅延やサーバータイムアウトなど「失敗は普通」と割り切り、反復を自動化する。

### 最低限のルール

> 真の TDD が難しいなら、せめてリグレッションテストから始める。

---

## 3. BDD にたどり着くまで

### E2E テストで気づいた限界

- E2E テストが軌道に乗り、リファクタリング後の動作確認サイクルが速くなった。
- しかし「エンドユーザーの実際の操作から紐付けてテストするとはじめて見つかる不具合」は、E2E の五月雨式打ちでは収束できなかった。

### BDD という選択肢

- 「ユースケースを書いたらテストできる？」と Copilot に尋ねたところ、**振る舞いからのテスト**が可能と回答。
- Claude にセカンドオピニオンを求め、**BDD（Behavior Driven Development）** の有効性を確認。
- エージェントを使った開発には、BDD という「からくりでモノを作る」フレームワークが最適と判断。

---

## 4. BDD の導入手順

### ステップ 1 — Copilot 全体のお約束を整備する

`.github/copilot-instructions.md` にプロジェクト全体のルールを記述する。

```md
## Scope

- Legacy PHP code analysis and incremental refactoring.
- PHP 8.5 compatibility updates with minimal behavioral regression.
- SQL, E2E, CI workflow, and security/documentation support.

## Coding Standards

- Keep existing behavior unless the task explicitly requests a change.
- Apply PSR-12 gradually; avoid large unrelated formatting-only diffs.
- Prefer explicit guards for legacy arrays and external input.
- Add concise comments only where logic is not self-evident.

## Testing and Validation

- Run E2E tests at meaningful work boundaries.
- Before E2E, confirm name resolution and host routing assumptions.
- Treat production-like data as sensitive and minimize exposure.
- For markdown changes, run markdownlint and fix violations.
```

### ステップ 2 — E2E 環境を選ぶ

| ツール | 採否 | 理由 |
|--------|------|------|
| Selenium | ✗ | ドライバ問題で動作せず |
| Playwright | ✗ | 同上 |
| **Puppeteer** | ✓ | 軽量で CI でも安定動作。Auth0 等の強固な認証は苦手だが、認証不要ページなら十分 |

### ステップ 3 — BDD テンプレートを Copilot に作らせる

以下のプロンプトで Gherkin テンプレートと instructions への追記を依頼した。

```
仕様書が存在しないため、「振る舞い」を記述して E2E テストにつなげたい。
BDD のテンプレート（Gherkin 形式）を作成してください。
また、.github/copilot-instructions.md にも必要事項を追記してください。
```

#### 生成された instructions への追記例

```md
### BDD テスト設計

- 仕様書が存在しないページは `tests/e2e/PAGE_INVENTORY.md` を参照し、
  `tests/e2e/BDD_TEMPLATE.md` のテンプレートに従ってシナリオを設計する。
- 読み取り専用テスト（表示確認）は production でも安全に実行可能。
- 書き込みテスト（フォーム送信）は `ALLOW_MUTATION=true TARGET_ENV=dev` でのみ実行。
- BDD テストの最初のテストケースは **weather-check**（環境到達確認）とする。
- テスト用 ID/PW はハードコードせず、管理画面からスクレイピングする。
- DB 状態を変更するテストは cleanup テストを末尾に配置し、状態を復元する。
- リファレンス実装: `tests/e2e/password_inquiry.bdd.test.js`
```

## 5. BDD 導入後の開発フロー

### Gherkin でシナリオを書く

普通の業務言葉でシナリオを記述するだけで、Copilot がテストコードに翻訳してくれる。

```gherkin
Feature: ブログにログインしてコメントを書き、メール通知が届くか確認する

  開発者がブログをテストし、メール通知の動作を確かめる。

  Scenario: ログインしてコメント送信ページに遷移する
    Given 開発者は *** というID と XXXX というPW で、所定の URL からログインする
    When  パスワードが正しい
    Then  ログインに成功し、次のページへ遷移できる
```

### 生成されるテストコードのイメージ

```js
// Scenario: テスト山テスト太郎さんの問い合わせ送信 [正常系]
//   Given  問い合わせフォームにアクセス済み
//   When   テストデータをフォームに入力し「次へ」をクリック
//   Then   確認画面に遷移し「送信」ボタンが表示される
//   And    入力値が確認画面に反映されている
//   When   「送信」ボタンをクリック
//   Then   「ありがとうございました」の完了メッセージが表示される
//   And    PHP エラーがない
```

### BDD 導入前後の比較

| 観点 | 導入前 | 導入後 |
|------|--------|--------|
| テスト指示方法 | 毎回プロンプトで細かく説明 | Gherkin シナリオを渡すだけ |
| 業務担当者との共有 | 難しい | 日本語シナリオで共有できる |
| テスト漏れの収束 | 五月雨式で収束しない | シナリオ網羅で体系化できる |

---

## 6. おわりに

レガシーアプリは怖い。「なぜこんな実装になっているのか」がどこにも書かれていないからだ。

**命名規則・フォルダ構成ルールを定めてからプロジェクトを始める** — これは当時の開発者には存在しなかった常識かもしれないが、今の開発では最初のステップとして欠かせない。

Copilot + BDD の組み合わせは、仕様書のないレガシー業務アプリのリファクタリングにこそ真価を発揮する。
